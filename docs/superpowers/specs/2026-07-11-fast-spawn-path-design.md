# UvEcs Fast Spawn Path — Design

## Контекст и мотивация

Ядро хранилища готово (151 тест + фаззер). Полный бенчмарк автора (18 участников)
показал устойчивую картину:

- **Итерация** — на уровне лидеров: `UvEcs_MonoThread` ≈ `Friflo_Mono`, быстрее обычного
  `Arch_Mono`, вдвое быстрее `DefaultEcs`, ноль аллокаций.
- **Создание** — слабое место: в 2–4× медленнее `Arch`/`Friflo`.

### Диагноз (подтверждён по коду)

Разрыв на create — архитектурный, а не про скан:

- `World.Create()` (`World.cs:40`) всегда кладёт сущность в `_archetypes[0]` — архетип
  **без компонентов**.
- Каждый `Add<T>` (`World.Structural.cs:5`) выполняет полный `Migrate`: вставка строки в
  приёмник + копирование растущей строки + swap-remove из источника.
- Итог: `Create(); Add(Position); Add(Velocity)` = **две миграции**; N-компонентная
  сущность = **N миграций**. Arch/Friflo вычисляют финальный архетип один раз и вставляют
  напрямую.

O(1)-кеш чанка со свободным местом (`Archetype._chunkWithSpace`, уже реализован, ждёт
коммита) снял вторичную цену — линейный скан чанков на каждую вставку — и дал 15–27% на
скан-тяжёлых кейсах. Но доминирует именно цепочка миграций; кеш её не трогает.

### Целевой юзкейс

Сетевой игровой сервер, тик 20 Гц. **Спавн — на горячем пути** (волны врагов, залпы пуль,
эффекты): много сущностей одного архетипа за тик. Значит, скорость create — реальный
приоритет, а не только цифра в бенчмарке. На горячем пути сущность обычно нужна **сразу**
(дописать состояние, получить Id), поэтому нужен **немедленный** быстрый create, а не
отложенный буфер.

## Цель

1. Прямое создание в финальном архетипе — **ноль миграций**.
2. Батч-спавн сущностей одного архетипа с одноразовым резервом чанков.
3. Закрыть разрыв к `Arch`: одиночный create — до ~1.5× (с 2–4×); `CreateMany` быстрее
   цикла одиночных create.

## Что НЕ входит (осознанно)

- **CommandBuffer / §7** (безопасная структурная мутация во время итерации) — задача
  Systems-плана. Он переиспользует direct-insert примитив этого плана; ничего не выкидываем.
  Наивный (отложить + проиграть по одному) буфер create не ускоряет — те же миграции;
  полезен только коалесцирующий вариант, а он большой и тянет Systems вперёд.
- **Публичный `Reserve`** — YAGNI. Резерв нужен только внутри `CreateMany`.
- **Bulk destroy.**
- **Починка `RemoveFromAllSparseSets`** — см. «Задокументированный следующий шаг».

## Публичный API (новое)

```csharp
// Прямое создание — финальный архетип за один шаг, ноль миграций.
Entity Create<T1>(in T1 c1);
Entity Create<T1,T2>(in T1 c1, in T2 c2);
Entity Create<T1,T2,T3>(in T1 c1, in T2 c2, in T3 c3);
Entity Create<T1,T2,T3,T4>(in T1 c1, in T2 c2, in T3 c3, in T4 c4);

// Батч-спавн — count сущностей ОДНОГО архетипа, чанки резервируются один раз.
void CreateMany<T1>(int count, Span<Entity> dest);
void CreateMany<T1,T2>(int count, Span<Entity> dest);
void CreateMany<T1,T2,T3>(int count, Span<Entity> dest);
void CreateMany<T1,T2,T3,T4>(int count, Span<Entity> dest);
```

- Все `Tn : unmanaged, IComponent`.
- Арность до 4: покрывает бенчмарк (One/Two/Three) + запас. Нужно больше — существующий
  путь `Create()` + `Add<T>` (миграции, но это уже вне горячего пути).
- Существующий `Create()` без аргументов остаётся без изменений.

## Механизм: прямое создание

`Create<T1,T2>(c1, c2)` повторяет примитивы `World.Create()`, но целится в **финальный**
архетип вместо `_archetypes[0]`, и **не вызывает `Migrate`**:

1. Собрать финальную `ComponentMask`: `mask.Set(ComponentType<T1>.Id)` и т.д.
2. `var to = GetOrCreateArchetype(in mask);` — одна словарная выборка.
3. `var e = Entities.Create();`
4. `var chunk = to.GetOrCreateChunkWithSpace(Pool, out int chunkIndex);` — O(1) через кеш.
5. `int row = chunk.AddRow(e);` — строка приходит занулённой (`AddRow` уже затирает).
6. Записать компоненты по ссылке: `chunk.GetRef<T1>(row) = c1;` и т.д.
7. Заполнить `EntityRecord` (`ArchetypeId`, `ChunkIndex`, `Row`).
8. `to.BumpStructuralVersion();`

Колонки под `T1..Tn` в `chunk` существуют, потому что `Layout` архетипа построен из той же
маски. Промежуточные архетипы (`{T1}`, `{T1,T2}`) не создаются.

## Механизм: батч-спавн (`CreateMany` + `Archetype.Reserve`)

`CreateMany` выносит из цикла две пер-сущностные цены, которые цикл одиночных `Create<T…>`
всё ещё платит: словарную выборку маска→архетип и рост числа чанков.

1. Проверка: `dest.Length >= count`, иначе `ArgumentException`. `count < 0` — тоже throw.
2. Собрать маску, `to = GetOrCreateArchetype(in mask)` — **один раз**.
3. `to.Reserve(Pool, count)` — новый internal-метод: пред-рентит достаточно чанков под
   `count` строк. Число строк на чанк = `Layout.Capacity` (`Chunk.Capacity`). Резерв
   гарантирует, что цикл заполнения не чередует вставку с аллокацией чанков.
4. Плотный цикл `count` раз: `Entities.Create()` → `GetOrCreateChunkWithSpace` (O(1)) →
   `AddRow` → записать `Entities.GetRecord` → `dest[i] = e`.
5. `to.BumpStructuralVersion()` один раз в конце (не на каждую строку).

Компоненты стартуют с default (строки занулены); значения ставит вызывающий после — по
свежезаписанным, кеш-горячим строкам. `count` может превышать вместимость одного чанка —
резерв и цикл спокойно охватывают несколько чанков.

`Reserve` остаётся internal. Публичного `world.Reserve(...)` нет, пока не появится сценарий
поэтапного спавна.

## Инварианты и edge cases

- **Дубликат типа в дженериках** (`Create<Position,Position>`) — в Debug бросаем (иначе
  вторая запись молча затирает первую в одной колонке). В Release — не проверяем (горячий
  путь; это баг вызывающего).
- **`dest` короче `count`** — `ArgumentException`, до любых мутаций.
- **`count == 0`** — валидно, no-op (ничего не резервируем, не создаём).
- **`count` > вместимости чанка** — несколько чанков, покрыто резервом и циклом.
- **Занулённая строка** — гарантирует `AddRow` (переиспользование буферов пула).
- **Structural version** — бампается (одиночный create — на сущность; `CreateMany` — один
  раз на пакет), чтобы §11-гард итерации ловил эти мутации.

## Тестирование

Unit (`tests/UvEcs.Tests`):

- `Create<T…>` кладёт сущность в правильный архетип, с правильными значениями и записью.
- **Ключевой тест: промежуточные архетипы не создаются** — `ArchetypeCount` растёт только на
  финальный архетип, а не на цепочку `{T1}`, `{T1,T2}`. Это доказывает, что цепочки миграций
  нет.
- Эквивалентность: `Create<T1,T2>(a,b)` даёт то же наблюдаемое состояние (компоненты,
  значения), что `Create()+Add(a)+Add(b)`.
- Гард дубликата типа (Debug).
- `CreateMany`: `count` сущностей в одном архетипе, все живы, Id различны, записи
  (чанк/строка) верны; кейс `count` > вместимости чанка (несколько чанков); `dest` короче
  `count` → throw; `count == 0` → no-op.
- **Фаззер** (`FuzzInvariantTests`): добавить `Create<T…>` и `CreateMany` как операции —
  проверяются против пяти структурных инвариантов.

## Валидация бенчмарком

- Перевести сценарии `CreateOne/Two/Three` (`bench/UvEcs.Bench`) на `Create<T…>()`.
- Добавить `CreateMany`-вариант (один архетип, пакетом).
- **Добавить отдельный замер destroy** — чтобы увидеть, стал ли destroy высоким столбом
  после ускорения create (арифметика долей). Не гадаем — меряем.
- Прогон `--job medium`, сравнение с `Arch`/`Friflo` на одной машине.
- Цель: одиночный create в пределах ~1.5× от `Arch`; `CreateMany` быстрее цикла одиночных
  create.

## Первый шаг плана

Закоммитить уже написанный O(1)-кеш чанка (`Archetype._chunkWithSpace`, 151/151 зелёный),
чтобы он не висел незакоммиченным под этой работой.

## Задокументированный следующий шаг (вне этого плана)

`Destroy` не имеет проблемы цепочки миграций: это **один** swap-remove из текущего архетипа
независимо от числа компонентов — разворачивать архетип назад никто не заставляет. Этот план
destroy-путь не трогает, регрессию не вносит.

Но `RemoveFromAllSparseSets` (`World.Sparse.cs:46`) на **каждое** удаление прогоняет цикл по
**всем** зарегистрированным sparse-типам, даже если у сущности sparse-компонентов нет. Это
O(число sparse-типов) на destroy. В текущем бенчмарке sparse-типов нет, поэтому цифры это не
двигает — потому починка и вне плана. Чистое решение: флаг «есть ли sparse-данные» в
`EntityRecord`; sparse-free сущности (общий случай) пропускают цикл, destroy становится
честным O(1). Ведут флаг `AddSparse`/`RemoveSparse`.
