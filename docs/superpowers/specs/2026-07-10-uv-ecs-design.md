# UvEcs — дизайн ECS для MMO на .NET 8

Дата: 2026-07-10
Статус: согласован, готов к планированию реализации

## 1. Задача

ECS-библиотека на C# / .NET 8 для MMO. Приоритет — скорость исполнения, не выразительность API.
Рефлексии нет ни на старте, ни в рантайме. Большинство сущностей в мире однотипны, поэтому
основное хранилище — архетипы.

Целевой масштаб одного мира (зоны/шарда): **~10 000 сущностей, десятки типов компонентов**.
Все числовые решения в этом документе откалиброваны под этот масштаб. При переходе к 100k
часть выводов меняется, и это отмечено в тексте.

Серверный тик: 20 Гц, бюджет 50 000 мкс.

## 2. Границы

Библиотека **не** делает:

- рендер и любую интеграцию с Godot — ECS не управляет нодами;
- интерес-менеджмент (AOI), приоритеты пакетов, транспорт — это код игры;
- managed-компоненты (`string`, `List<T>`, ссылки на объекты) — только `unmanaged`;
- предсказание чего-либо, кроме сущностей, помеченных тегом `Predicted`;
- полный детерминированный rollback мира.

Godot остаётся движком клиента, но ECS на клиенте **не нужен**: клиент предсказывает одну
сущность, и для этого достаточно вызвать общую чистую функцию (см. §10).

## 3. Пакеты

```
UvEcs.Abstractions   IComponent / ITag / ISparse, атрибуты   ← клиент и сервер
UvEcs.Shared         компоненты игры + чистая логика систем   ← клиент и сервер
UvEcs               ядро: World, архетипы, чанки, query       ← сервер
UvEcs.Codegen       Roslyn-генератор (net8.0)                 ← время сборки
UvEcs.Tests
```

Зависимости строго в одну сторону. Ядро не ссылается ни на что, включая GodotSharp, — это даёт
headless-сервер и тесты без движка.

Клиент тянет `Abstractions` + `Shared`. Ни строчки ядра.

`UvEcs.Codegen` таргетит **net8.0**, а не netstandard2.0: `netstandard2.0` нужен только для
загрузки анализатора внутри Visual Studio (процесс на .NET Framework). VS не поддерживается,
рабочие среды — VS Code и Rider, оба на .NET Core. Проверено: генератор на net8.0 загружается
компилятором при `dotnet build` и отрабатывает.

Отдельного пакета `UvEcs.Godot` нет. Конверсия `float3 ↔ Vector3` — пять строк в игровом проекте.

## 4. Модель данных

### Сущность

```csharp
public readonly struct Entity { public readonly int Id; public readonly uint Version; }
```

Generational index. Таблица `EntityRecord[]` индексируется `Id`: архетип, номер чанка, строка,
версия. При удалении версия инкрементится, слот уходит во free-list. 10k сущностей = 160 КБ.

### Три хранилища, выбор — на этапе компиляции

```csharp
public interface IComponent { }   // колонка в чанке архетипа
public interface ITag { }         // бит в маске чанка
public interface ISparse { }      // sparse set
```

API разведён по констрейнтам: `Add<T>() where T : unmanaged, IComponent`,
`SetTag<T>() where T : unmanaged, ITag`. Положить тег в чанк — ошибка компиляции.

Маркерный интерфейс **не стоит ничего в рантайме**. Констрейнт живёт в метаданных generic-параметра.
Измерено: IL двух одинаковых методов, с констрейнтом `IComponent` и без, побайтово идентичен
(`[2 40 1 0 0 43 78 42]`). Интерфейсных вызовов нет — маркер пуст, девиртуализировать нечего.

`IManaged` **отсутствует намеренно**. Он вводился ради ссылки на `Godot.Node`; раз ECS не управляет
нодами, обоснование исчезло. Строки — `FixedString32` (inline-буфер). Следствия: rollback это
`memcpy`, сериализация это `memcpy`, GC не видит содержимое ECS (`byte[]` не содержит ссылок —
`IsReferenceOrContainsReferences<byte>() == false`, проверено).

### ComponentId

`ComponentType<T>.Id` — `static readonly int`, выдаётся при первом обращении. JIT в tier1 сворачивает
такое поле в константу, поэтому `const` не даёт выигрыша. Идентификатор стабилен внутри процесса и
**нестабилен между процессами**, поэтому наружу — в сеть и в файлы — не выходит никогда. Там работает
`NetId` (§9).

Теги нумеруются в отдельном пространстве: `TagType<T>.Bit` — `static readonly ulong`, равен
`1UL << index`. Компоненты и теги не делят номера, потому что живут в разных структурах.

### Маска компонентов

```csharp
[InlineArray(4)]
public struct ComponentMask { private ulong _e0; }   // 32 байта, 256 компонентов, ноль аллокаций
```

Категорийные маски (пары «категория + биты») отвергнуты: матчинг 1000 архетипов плоской маской —
**1.07 мкс**, категорийной — **9.25 мкс**. Косвенность дороже компактности. К тому же матчинг не в
горячем пути (см. §6).

## 5. Чанк и архетип

### Геометрия

Чанк — **16 КБ**, память: `GC.AllocateUninitializedArray<byte>(16384 + 64, pinned: true)` (Pinned
Object Heap). Массив закреплён навсегда, указатель берётся без `fixed`, `Dispose` не нужен.

POH даёт выравнивание только на 8 байт (измерено: из 200 массивов на границу 64 попали 25, типичный
адрес `& 63 == 40`). Поэтому указатель сдвигается внутри массива до границы 64 — массив закреплён,
адрес не поедет. Отсюда `+64` в размере.

Выигрыша от выравнивания на x86 нет: сумма 4096 `float` через `Vector256`, пять раундов после общего
прогрева — выровненный POH 30.5 мкс, `AlignedAlloc` 30.5 мкс, невыровненный со сдвигом на 8 байт
(пересекает кеш-линии) 30.1 мкс. Разница в пределах шума. Выравнивание делается как страховка для
ARM64, где измерить не удалось. Цена — 64 байта из 16 384.

`NativeMemory.AlignedAlloc` отвергнут: даёт то же выравнивание ценой ручного времени жизни.
Утверждение «аллокация не бросает исключений» неверно — проверено, бросает `OutOfMemoryException`.

Раскладка — SoA. Первые две колонки служебные:

```
[ header ][ Entity × Cap ][ ulong tags × Cap ][ Position × Cap ][ Velocity × Cap ] ...
```

Колонка `Entity` обязательна и используется тремя механизмами: swap-remove должен обновить
`EntityRecord` у строки, переехавшей на освободившееся место; `repVersion` индексируется по
`entityId`; итерация с доступом к `Entity` не должна ходить в другую структуру.

Ёмкость считается один раз при создании архетипа: `Cap = (16384 − header) / (8 + 8 + Σ размеров)`,
где первые два слагаемых — `Entity` и маска тегов. Для сущности с `Position` (12), `Velocity` (12)
и `Health` (4) это 44 байта, `Cap = 369`, и 10k сущностей занимают **28 чанков = 448 КБ**.
Заголовки массивов — 0.15% оверхеда.

Компонент больше 16 КБ — исключение при создании архетипа.

### Пул

Все чанки одного размера, поэтому **пул один на мир** и чанки взаимозаменяемы между архетипами.
Это отдельный аргумент за байтовый размер против «фиксированного числа сущностей»: там чанки
разных архетипов имеют разный размер и пул пришлось бы дробить.

Пустой чанк возвращается в пул с гистерезисом — архетип держит один пустой про запас, иначе
сущность, прыгающая через границу чанка, устраивает пинг-понг. Память ОС не отдаётся.

### Теги

Тег **не входит** в идентичность архетипа. Колонка `ulong` на сущность, 64 тега (расширяемо до
`[InlineArray(2)]`). `SetTag`/`UnsetTag` — один бит, без миграции.

Чанк кеширует `TagUnion` — OR всех масок. Если `(TagUnion & required) != required`, тега нет ни у кого
и чанк пропускается целиком. Обновляется как `Union |= mask` при установке бита и при вставке сущности;
при снятии не сужается. Оценка консервативна: может быть шире правды, и тогда мы лишь не пропустим
чанк, который могли бы. Точное значение пересчитывается в конце тика для чанков с флагом «маска
менялась» — это OR по колонке, микросекунды.

### Как выбрать хранилище

Решение определяется двумя вопросами: **есть ли у признака данные** и **сколько носителей**.
Редкость сама по себе хранилища не определяет — это частая ошибка.

**1. Различие неизменяемое** (`IsPlayer`, `IsNpc`, `IsProjectile`). Не заводится ничего: такое
различие почти всегда **уже выражено набором компонентов**. У игрока есть `Inventory` и `Account`,
у NPC — `AiState`. `Query<Inventory>` и есть «все игроки».

Заводить zero-sized компонент ради маркера — плохая идея: пустой `struct` в C# занимает **1 байт**
(`Unsafe.SizeOf<IsPlayer>() == 1`, неотличим от `struct { byte b; }`), то есть колонку в байт на
сущность. Пять таких маркеров срезают `Cap` с 369 до 331 (−10.3%) — примерно столько же, сколько
стоил бы переход на 128 тегов. Механизма `IZeroSized` в библиотеке нет намеренно.

**2. Флаг без данных, переключается** (`InCombat`, `Stunned`, `Rooted`, `Silenced`, `Dead`,
`Invulnerable`, `Mounted`, `Stealthed`, `Predicted`) — **тег**.

Тег стоит **один бит из 64 и ноль байт памяти**: колонка масок уже присутствует в каждом чанке,
установлен тег хоть у кого-нибудь или нет. Поэтому **редкость значения не имеет** — `Stealthed`
у трёх сущностей из десяти тысяч бесплатен ровно так же, как `InCombat` у половины.

Единственный дефицитный ресурс здесь — биты, а не байты.

**3. Данные, носителей больше ~25%** — обычный архетипный `IComponent`.

**4. Данные, носителей меньше ~25%** (`GuildBuff` с полями `Id`/`Until`) — `ISparse`, см. порог в §6.

**5. Биты кончились** — выселяется в `ISparse` тот тег, у которого **меньше всего носителей**:
sparse-драйвер выигрывает именно на малой доле (§6). Это выпускной клапан, а не первый выбор:
sparse-набор стоит `int[]` размером с число сущностей (40 КБ на 10k) плюс плотные массивы, и теряет
чанковую итерацию. Менять сорок килобайт и скорость запроса на один сэкономленный бит имеет смысл,
только когда бит действительно последний.

### Почему 64, и почему это обратимо

Маска тегов **никогда не пишется как `ulong`**. Везде — снаружи и внутри чанка, query, матчинга —
используется:

```csharp
public readonly struct TagMask {
    private readonly ulong _bits;                       // единственное место, где встречается ulong
    [MethodImpl(AggressiveInlining)] public bool HasAll(TagMask r) => (_bits & r._bits) == r._bits;
    // HasAny, Or, Equals ...
}
```

`TagType<T>.Bit` возвращает `TagMask`, не `ulong`.

Обёртка бесплатна, и это не метафора. `struct` в C# не имеет заголовка объекта и не размещается в
куче: `TagMask` в памяти — те же восемь байт, что `ulong`, а `TagMask[]` уложен байт в байт как
`ulong[]` (`Unsafe.SizeOf<TagMask>() == 8`). `HasAll` крошечный и помечен `AggressiveInlining`,
поэтому JIT подставляет его тело в место вызова, и от `mask.HasAll(req)` остаётся `(bits & req) == req` —
те же две машинные инструкции.

Измерено (25 раундов, чередование порядка, медиана): голый `ulong` 4.49 мкс, `TagMask.HasAll`
4.45 мкс — разница 1% при собственном разбросе 27%, то есть неотличимо.

Благодаря этому переход на 128 тегов — правка **одного файла**: поле меняется на `[InlineArray(2)]`,
пять методов переписываются, остальной код не замечает, потому что он никогда и не знал, что внутри
лежит `ulong`.

Неизбежная цена такого перехода — не рефакторинг, а ёмкость:

- проверка маски в цикле: 4.18 → 7.51 мкс на 10k (1.80x). Это 3.3 мкс = 0.0067% тика, неважно;
- **ёмкость чанка: 369 → 312 сущностей (−15.4%)**, чанков на 10k: 28 → 33. Восемь лишних байт на
  сущность платят все системы, включая не касающиеся тегов.

Поэтому 64 берутся с обоснованием, а не «пока хватает». Если список подойдёт к 64, сначала
перебираются корзины: редкий тег переезжает в `ISparse` (порог 25%, §6), неизменяемый оказывается
не нужен, потому что выражен набором компонентов. «Стало 65» означает «пора пересмотреть
категоризацию», а не «переписать движок».

### TagIntersection

`TagIntersection` (AND всех масок) **отвергнут**. Он экономил внутри цикла одну инструкцию `AND` и
сравнение на сущность по уже загруженной кеш-линии — около такта, ~3 мкс на 10k. При этом требовал
инвариант, который легко нарушить: правило «обновлять только в расширяющую сторону» не покрывало
вставку сущности в чанк, из-за чего маска могла соврать в опасную сторону («тег есть у всех», когда
его нет). Оптимизация меньше шума, риск реальный.

### Архетип

Отсортированный набор `ComponentId`, `ComponentMask`, смещения колонок, список чанков и граф переходов:

```csharp
Dictionary<int, Archetype> addEdges;      // componentId → архетип
Dictionary<int, Archetype> removeEdges;
```

`Add<Velocity>` не пересобирает множество и не ищет архетип по хешу — идёт по ребру за один lookup.
Рёбра строятся лениво. Поскольку большинство сущностей однотипны, после прогрева зоны граф перестаёт
расти.

Архетипы **никогда не удаляются**: граф переходов и кеши запросов держат на них индексы. Пустой
архетип — это маска, смещения и пустой список, порядка 200 байт; даже 500 архетипов это 150 КБ.
Освобождаются чанки, а не архетипы.

### Миграция

`Add`/`Remove` — это явные операции: swap-remove строки из чанка старого архетипа, вставка в чанк
нового, обновление `EntityRecord`. Одно копирование строки. Сущность в каждый момент находится ровно
в одном архетипе.

Sparse-наборы и таблица `repVersion` при миграции **не трогаются**: они индексируются по `entityId`,
а не по позиции в чанке. Порядка обновления не существует, потому что обновления нет.

## 6. Query

Долгоживущий объект, создаётся один раз при инициализации системы. Создание query в тике — главный
способ убить эту библиотеку.

```csharp
readonly Query _movable = world.Query().All<Position, Velocity>().None<Frozen>().Build();
```

Матчинг архетипа — три операции над `ComponentMask`:

```csharp
(arch.Mask & all) == all && (arch.Mask & none) == 0 && (any == 0 || (arch.Mask & any) != 0)
```

Кеш архетипов инвалидируется по счётчику `World.ArchetypeCount`. Если счётчик вырос, доматчиваются
**только новые** архетипы с запомненной позиции. Полного пересканирования не бывает. После прогрева
проверка вырождается в сравнение двух `int` — поэтому матчинг не в горячем пути.

### Два режима итерации

**По чанкам** — основной, без делегатов и диспетчеризации:

```csharp
foreach (ref readonly var chunk in _movable) {
    var pos = chunk.GetWrite<Position>();     // Span<T>
    var vel = chunk.GetRead<Velocity>();      // ReadOnlySpan<T>
    for (int i = 0; i < chunk.Count; i++) pos[i].Value += vel[i].Value * dt;
}
```

`GetWrite<T>()` штампует `repVersion[T]` для **всех `Count` строк чанка** сразу, при выдаче `Span`.
Перехватить запись в `Span` без накладных нельзя, поэтому иначе и не получится. Для систем, которые
проходят по всему чанку — а это подавляющее большинство, — это не консерватизм, а правда: система
движения действительно меняет `Position` у всех.

`GetRead<T>()` не штампует ничего.

`GetWriteNoVersion<T>()` + `chunk.MarkChanged<T>(row)` — для систем, пишущих одну сущность из
пятисот (типично: применение урона). Здесь штамп ставится ровно на изменённые строки.

Оба варианта нужны: без первого систему движения пришлось бы писать через индексатор со штампом на
каждом присваивании, без второго система урона пометила бы весь чанк.

**По сущностям** — включается, когда запрос требует sparse-компонент через `AllSparse<T>()`. Драйвером
становится наименьший из обязательных sparse-наборов; для каждого носителя берётся чанк и строка из
`EntityRecord`. Чанковый API в этом режиме недоступен — сущности размазаны по чанкам, `Span` на них
не натянуть. Проверяется в Debug.

`OptionalSparse<T>()` оставляет драйвером чанки, а lookup идёт внутри цикла. Это выбор пользователя
между двумя режимами, сделанный формой запроса.

### Когда sparse-драйвер перестаёт выигрывать

Измерено. Зависит от **ширины запроса**, потому что sparse-драйвер делает по одному разбросанному
чтению на каждую колонку.

Запрос из одного компонента (`Position`), 100k сущностей:

| носителей | sparse-драйвер | чанк + фильтр | |
|---|---|---|---|
| 1% | 1.37 мкс | 56.3 мкс | sparse 41x |
| 5% | 9.9 мкс | 76.9 мкс | sparse 7.7x |
| 50% | 77.6 мкс | 375 мкс | sparse 4.8x |
| 100% | 156 мкс | 99.2 мкс | чанки 0.64x |

Запрос из восьми компонентов, 100k сущностей:

| носителей | sparse-драйвер | чанк + фильтр | |
|---|---|---|---|
| 1% | 25.8 мкс | 411 мкс | sparse 16x |
| 25% | 624 мкс | 1018 мкс | sparse 1.6x |
| **50%** | **1950 мкс** | **1139 мкс** | **чанки 0.58x** |

**Правило: `ISparse` окупается, пока носителей меньше ~25% от того, что матчит остальная часть
запроса.** Дальше компонент переводится в архетип или в тег. Debug-сборка предупреждает при
превышении порога.

Побочное наблюдение: чанковый фильтр работает **хуже всего именно на 50%** (375 мкс против 99 мкс на
100%) — предсказатель ветвлений максимально ошибается. Утверждение «чанковый проход это предсказуемый
линейный доступ» неверно: фильтрация по sparse требует случайного чтения `sparseIdx[entityId]` на
каждой сущности.

## 7. Системы, command buffer, тик

### Что структурно, а что нет

Структурны только `Add`, `Remove`, `Create`, `Destroy` — то, что двигает сущность между архетипами.

`SetTag` / `UnsetTag` / `Set<T>` структурными **не являются**: тег это бит в колонке, архетип не
меняется, чанк не переезжает. Применяются немедленно прямо в цикле. Обязанность одна — пометить чанк
флагом «маска менялась».

Sparse `Add`/`Remove` архетип не меняют, но ломают итератор, если этот набор сейчас драйвер. Ловится
в Debug.

### Буфер

```csharp
EntityHandle h = ecb.Create();          // отдельный тип, не конвертируется в Entity неявно
ecb.Add(h, new Velocity(...));
ecb.Destroy(e);
```

Ключ сортировки команды — **`chunkIndex` позиции записи**, а не свойство сущности. Итератор
проставляет его в буфер перед вызовом делегата на чанк; вне параллельного цикла ключ равен нулю.
Внутри чанка работает ровно один поток, поэтому все его команды лежат в одном сегменте в порядке
записи, и **стабильная** сортировка по `chunkIndex` этот порядок сохраняет. Глобальный счётчик через
`Interlocked` не нужен — именно он убил бы параллельную запись.

Временная сущность, созданная внутри чанка, наследует его ключ, поэтому `Create` и последующие `Add`
применяются подряд. Таблица `EntityHandle.Index → Entity` (массив, не словарь) разрешает ссылки.

Цепочка `Create` + `Add` + `Add` — это N миграций. Горячий путь спавна:

```csharp
readonly ArchetypeHandle _projectile = world.CreateArchetype<Position, Velocity, Damage>();
ecb.Create(_projectile, new Position(p), new Velocity(v), new Damage(d));   // ноль миграций
```

Вложенные буферы не поддерживаются.

Переполнение временных `Id` недостижимо: счётчик сбрасывается при каждом playback, то есть после
каждой системы.

### Порядок тика

```
для каждой стадии:
    для каждой системы:
        system.Run(ctx)
        system.Ecb.Playback()          ← следующая система видит результат
    пересчитать TagUnion у помеченных чанков
    world.Tick++
```

Playback после **каждой** системы — это выбор семантики, не простоты. Порядок систем означает порядок
причинности: если система A создаёт снаряд, а система B в той же стадии двигает снаряды, B его
увидит. При playback в конце стадии — не увидела бы. Цена — один `if (count == 0) return;` на пустой
буфер.

`world.Tick` инкрементится в самом конце, поэтому все записи одного тика штампуются одним номером.

Отложенный `Destroy` означает, что данные удаляемой сущности будут прочитаны и записаны до конца
тика. Это корректно и является частью контракта.

## 8. Параллелизм

Явный, на уровне вызова, с порогом:

```csharp
_heavyQuery.ForEachChunkParallel(minChunks: 8, static (in Chunk c, in Ctx ctx) => { ... });
```

Меньше `minChunks` — молча в один поток. Делегат зовётся раз на чанк (28 раз при 10k), его стоимость
теряется на фоне сотен сущностей внутри.

**Параллелизм не является режимом по умолчанию.** При 10k сущностей система, читающая `Velocity` и
пишущая `Position`, занимает 12–13 мкс, а запуск `Parallel.For` стоит 5–20 мкс накладных. Окупается
только тяжёлая работа на сущность: поиск пути, широкая фаза столкновений, AI, видимость.

Гонок нет по построению: чанк принадлежит одному потоку, `world.Tick` инкрементится вне параллельных
секций, `EntityRecord` пишется только при playback (однопоточно), `TagUnion` пересчитывается после
стадии. Единственный риск — нарушение контракта: `world.GetRef` по произвольной сущности из
параллельного цикла. Ловится флагом `_inParallelChunk` под `#if DEBUG`; проверять
`ManagedThreadId` на каждом чтении слишком дорого.

## 9. Сеть

Ядро даёт сети ровно две вещи: таблицу версий и доступ к байтам строки. Если в ядре появится слово
`player` — значит что-то сделано не так.

### repVersion вместо версий чанков

**Версий на чанках нет.** Они были в первом варианте дизайна и оказались бесполезны: дельта в MMO
считается по области интереса игрока — две-три сотни сущностей, разбросанных по разным чанкам. Чанк
почти всегда грязный, потому что в нём лежит кто-то ещё.

Вместо этого — side-таблица по сущности:

```csharp
uint[] repVersion;   // [componentId][entityId] = тик последней записи
```

Дельта для игрока с `ack = A` — обход его сущностей и проверка `repVersion[c][e] > A`. Ни снимков,
ни baseline, ни побайтового сравнения. Версия монотонна, поэтому **лимита на глубину истории нет**:
игрок, отставший на десять секунд, получит корректную дельту.

Цена: 234 КБ на 10k сущностей при шести реплицируемых компонентах, плюс один store при записи.
Измерено: движение 10k сущностей 12.76 мкс без штампа, 19.55 мкс со штампом — **+53% к системе, но
+6.8 мкс из 50 000 мкс тика, то есть 0.014%**.

Версии **не кладутся колонкой в чанк**: шесть реплицируемых компонентов срезают ёмкость с 369 до 239
сущностей (−35%), замедляя все проходы всех систем ради сети.

**Кольцо снимков на 32 кадра отвергнуто.** Оно занимало до 14 МБ, требовало пула, вводило лимит
истории и решало задачу рассылки всего мира, которой у нас нет.

### Сериализация

Компоненты `unmanaged` и `[StructLayout(LayoutKind.Sequential, Pack = 1)]`, поэтому сериализация —
`MemoryMarshal.Write`, то есть `memcpy`.

`UvEcs.Codegen` собирает из `[NetComponent(NetId = 17)]` таблицу `netId → (localId, размер)` и
проверяет на этапе компиляции уникальность номеров и blittable-ность. `NetId` задаётся руками и
является контрактом протокола: добавление компонента не сдвигает чужие номера. Дубль `NetId` иначе
всплыл бы как рассинхрон у игроков.

### Транспорт (принадлежит игре, зафиксирован для контекста)

LiteNetLib 2.1.4 + MessagePack 3.1.8.

Граница проходит по каналу доставки:

- **состояние мира** (дельты) — `Unreliable`, сырой бинарь. Схема с `ack` самоисправляется: потеря
  пакета не двигает `ack`, следующий пакет содержит всё с последнего подтверждённого тика.
  Ретрансляция не нужна;
- **события и команды** (логин, чат, крафт) — `ReliableOrdered`, MessagePack;
- **ввод игрока** — `Unreliable` с дублированием нескольких последних вводов в каждом пакете.
  `Reliable` противопоказан: пересылка старого ввода приедет позже, чем нужна.

Измерено: MessagePack на `float` даёт +33% (`Position` 12 → 16 байт, каждый `float32` кодируется
5 байтами), на малых `int` **выигрывает** у сырых байт (`Health` 8 → 7). Суммарно на дельте +12%.
Рефлексия не обязательна: `MessagePack.SourceGenerator` подключается автоматически.

Жёсткое ограничение: `NetConstants.MaxUnreliableDataSize = 1431`, `InitialMtu = 1024`, и
`Unreliable` **не фрагментируется**. При 25 байтах на сущность (`entityId` + маска + `Position` +
`Health`) в пакет влезает ~57 сущностей. В плотной толпе их двести, и все двигаются.

Поэтому игре нужен бюджет пакета и накопитель приоритета: сущности сортируются по приоритету, в пакет
пишутся, пока есть бюджет, не влезшие копят приоритет. В бою растёт не пакет, а задержка обновления
дальних целей. Это код игры, не библиотеки.

Квантование позиции (три `int16` по сетке мира) сожмёт сущность с 25 до 11 байт и удвоит плотность
пакета. Отложено до появления профиля трафика.

## 10. Предсказание и rollback

Предсказываются только сущности с тегом `Predicted` (бит в маске, ставится и снимается бесплатно).
Питомец игрока тоже помечается `Predicted` и работает.

**ECS на клиенте не используется.** Логика предсказуемых систем вынесена в чистые функции:

```csharp
// UvEcs.Shared — знает про компоненты, не знает про World
public static class MovementLogic {
    [MethodImpl(MethodImplOptions.AggressiveInlining)]
    public static void Step(ref Position p, in Velocity v, in MoveInput inp, float dt) { ... }
}
```

Сервер оборачивает её системой и гоняет по чанкам. Клиент зовёт напрямую над своей сущностью. Это
буквально одна функция, один IL — отсюда и берётся идентичность предсказания, а не из общего ECS.

Цена — потеря векторизации на сервере. Измерено на 10k сущностей: чистая функция поштучно 12.27 мкс,
SoA + AVX256 2.09 мкс. SIMD быстрее в **5.9 раза** и экономит **10 мкс из 50 000 (0.02% тика)**.
Не окупает ни одной строчки сложности. SIMD остаётся там, где он нужен и где ничего не предсказывается:
широкая фаза столкновений, видимость, AI.

Выдрать в чистую функцию можно только **локальную** систему: читает и пишет одну сущность плюс
неизменяемый контекст, не ходит за другими сущностями, не делает структурных изменений. Это не
совпадение — предсказать на клиенте можно ровно то, что локально, потому что достоверного состояния
чужих сущностей у клиента нет. Множества «выдираемых» и «предсказуемых» систем совпадают.

Каст с поиском целей в радиусе не предсказывается. Лечение союзника не предсказывается.

### Откат

Клиент держит два кольца по 64 тика: свои вводы и снимки **своей** сущности (`unmanaged` struct).
Пришло подтверждение на тик `T` — сравниваем со снимком `states[T % 64]`. Разошлось — восстанавливаем
из серверного состояния и переигрываем вводы с `T+1`. При пинге 100 мс и 20 Гц это два-три прогона
над одной сущностью.

Контракт: предсказанная система читает и пишет только сущности с тегом `Predicted`. Иначе откат не
вернёт то, что она тронула. Проверяется в Debug.

**Битовой идентичности float между клиентом и сервером не будет.** Один и тот же C# на x86 и ARM64
даёт разные результаты: JIT волен свернуть умножение со сложением в FMA, а FMA не округляет
промежуточный результат. Поэтому:

- сравнение по умолчанию побайтовое;
- допуск задаётся явно и только там, где клиент сам считает float:
  `[Predicted(Tolerance = 1e-4f)]` на `Position` и `Velocity`.

Ошибка не накапливается — сервер шлёт коррекцию каждый тик.

Запрещать FMA и фиксировать порядок операций ради битового детерминизма дороже, чем допуск, и не
нужно: сервер авторитет.

## 11. Ошибки

**Ловит компилятор.** Класс хранилища — через констрейнты интерфейсов.

**Ловит Debug.**

- структурное изменение во время `foreach` — через `Archetype.StructuralVersion`;
- `world.GetRef` внутри параллельного цикла — через флаг `_inParallelChunk`;
- `Chunks()` на запросе с `AllSparse`;
- sparse `Add`/`Remove` набора, который сейчас драйвер;
- sparse-набор превысил 25% кандидатов запроса — предупреждение.

**Проверяется всегда, включая Release.** Протухшая `Entity`: `GetRef` сравнивает версию из
`EntityRecord` с версией в дескрипторе. Одно сравнение по уже загруженной кеш-линии, предсказуемая
ветвь, около такта. Молча читать чужую память недопустимо.

**Не ошибка.** `Destroy` уже удалённой сущности — no-op. Две системы в одном тике могут независимо
решить, что моб умер.

**Ошибка среды.** OOM при аллокации чанка пробрасывается.

## 12. Тестирование

Юнит-тесты на изолированные структуры: граф переходов, sparse set, маска, расчёт ёмкости.

**Фаззинг с проверкой инвариантов** — основной инструмент. Случайная последовательность операций,
после каждой проверяется:

- каждая живая сущность лежит ровно в одном чанке;
- `EntityRecord[e]` указывает на чанк и строку, где записана та же `e` (обратная ссылка сходится);
- `chunk.Count` равен числу занятых строк;
- `TagUnion` — надмножество OR всех масок чанка;
- `sparse.dense[sparse.sparse[e]] == e` для каждого носителя.

Ловит swap-remove, забывший обновить `EntityRecord`, миграцию, потерявшую тег, sparse set с висячим
индексом.

**Детерминизм playback.** Одна последовательность команд, записанная из разных потоков в разном
порядке, обязана давать побитово одинаковый мир. Сто прогонов, сравнение хеша чанков.

**Golden-тест идентичности логики.** Одна сущность прогоняется через серверный путь (система по
чанку) и через клиентский (прямой вызов `MovementLogic.Step`), результаты сверяются бит в бит. Пока
реализация одна, тест зелёный по построению; он покраснеет в день появления второй.

## 13. Бенчмарки

**Не в CI.** Бенчмарк на общем раннере даёт шум в десятки процентов: в ходе этого дизайна непрогретый
JIT показал «невыровненный доступ на 68% быстрее выровненного», что оказалось артефактом. Запуск
руками перед мержем крупных изменений, базовые числа зафиксированы:

| сценарий | измерено |
|---|---|
| итерация 10k × 2 компонента | 12.8 мкс |
| то же + штамп repVersion | 19.6 мкс |
| матчинг 1000 архетипов | 1.07 мкс |
| sparse-драйвер, 1% носителей, 100k | 1.37 мкс |
| SoA + AVX256, 10k | 2.09 мкс |
| миграция архетипа (add компонента) | не измерено |
| create/destroy 10k | не измерено |

### Методика (не пропускать)

За время проработки дизайна бенчмарк выдал **перевёрнутый результат трижды**: «невыровненный доступ
на 68% быстрее выровненного», «128-битная маска быстрее 64-битной», «обёртка над `ulong` стоит
денег». Каждый раз причина одна — шум харнесса больше измеряемого эффекта.

Обязательные правила:

1. **Общий прогрев всех вариантов вместе**, десятки тысяч итераций, до замеров. Непрогретый tier0-JIT
   искажает первый измеренный вариант.
2. **Порядок вариантов чередуется** между раундами. Первый замер систематически хуже.
3. **Не меньше 25 раундов**, берётся медиана, а не одно число.
4. **Печатается собственный разброс** (min/med/max) каждого варианта. Если разница между вариантами
   меньше разброса внутри варианта — разницы нет, независимо от того, как красиво смотрятся числа.
5. **Проверяется, что варианты делают одну работу** (одинаковое число совпадений, одинаковая сумма).
   Иначе сравниваются разные алгоритмы.
6. Одинаковый стиль доступа к памяти в обоих вариантах (`Span` + `ref` либо индексатор, но не
   вперемешку).

Типичный собственный разброс на этой машине — 27–35%. Эффекты меньше этого порога не измеряются
таким харнессом вообще, и для них нужен либо счётчик инструкций, либо сравнение IL/asm.

## 14. Отвергнутые решения

| решение | причина отказа |
|---|---|
| версии на чанках | дельта идёт по AOI, чанк почти всегда грязный. Заменено на `repVersion` по сущности |
| кольцо снимков на 32 кадра | 14 МБ, пул, лимит истории — всё это решало задачу рассылки мира, которой нет |
| `TagIntersection` | экономит ~такт на сущность, требует инварианта, который легко нарушить |
| `IManaged` | вводился ради `Godot.Node`; ECS не управляет нодами |
| `netstandard2.0` для генератора | нужен только для Visual Studio, которая не поддерживается |
| категорийные маски компонентов | 8.7x медленнее плоской, и матчинг не в горячем пути |
| пакет `UvEcs.Godot` | остаётся пять строк конверсии, им место в игре |
| AOI и приоритеты в библиотеке | интерес в MMO не сводится к радиусу (пати через карту, рейд-босс, невидимка) |
| SIMD в предсказуемых системах | экономит 0.02% тика, ломает идентичность клиент/сервер |
| параллелизм по умолчанию | при 10k накладные `Parallel.For` сопоставимы с работой |
| `NativeMemory.AlignedAlloc` | то же выравнивание ценой ручного времени жизни |
| `BeginWrite`-скоуп с бампом «при реальной записи» | `memcmp` на выходе стоит столько же, сколько дифф, который он экономит |
| playback в конце стадии | меняет семантику: система не увидит сущность, созданную соседней системой |

## 15. Открытые вопросы

- Не измерено на ARM64: выигрыш от выравнивания чанка. Если его нет и там, сдвиг на 64 можно убрать
  (одна строка в аллокаторе).
- Не измерены: стоимость миграции архетипа и `create`/`destroy` 10k. Нужны до фиксации базовых чисел.
- Число тегов зафиксировано на 64 (`ulong`) с обоснованием в §5. Остаётся выписать реальный список
  второй корзины до начала реализации: если он превысит ~40, стоит пересмотреть решение, пока код
  не написан.
- Квантование позиции — после профиля трафика.
